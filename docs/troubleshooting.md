# 観測されやすい誤検出と対応

このページは、`nihongo-slopless` を運用したときに見られやすい指摘の癖と、その読み解き方をまとめたものです。指摘は編集候補であり、文章の正解判定でも、書き手の属性判定でもありません。

ここに出てくる観測例は、公開文書に載せるための合成例です。第三者の記事、個人原稿、ローカル検証データからの抜粋ではありません。実運用で得た本文や検証データは、権利と公開範囲を確認したうえでローカルに保持してください。

各エントリは次の観点で読みます。

- 観測例: 起きやすい指摘を示す合成例
- 起きやすい理由: ルールが反応する表層条件と、文脈上のずれ
- 取りうる対応: 修正、設定調整、理由付き無視コメントのいずれか

ルールIDは `nihongo-slopless/<rule>` の短縮形で示します。

## `absolute-claim`

### 量的な `100%`

- 観測例: `契約書では、稼働率100%を目標値として置く。` (`absolute-claim`)
- 観測例: `この制度では、対象費用の100%を補助する。` (`absolute-claim`)

`100%` は強い断定のマーカーになりやすい一方で、契約、会計、政策文書では単なる割合表記としても使われます。数値が条件や制度の一部であれば、文章上の過剰な断定とは限りません。

対応:

- 数値表記が多い文書では、`absolute-claim` の重要度を `info` に下げる。
- 数値の意味が曖昧なら、`何に対する100%か` を補う。
- そのまま残す場合は、理由付き無視コメントで「制度上の割合表記」などと記録する。

### 表現そのものを題材にした文

- 観測例: `見出しに「必ず成功する」と書くと、反証の余地が狭くなる。` (`absolute-claim`)
- 観測例: `教材では、「完全に」という副詞をどう扱うかを確認する。` (`absolute-claim`)

強い語を批評・教材・引用として扱っている文では、その語が本文の主張ではなく題材です。引用符や文脈で題材化が明確なら、修正しない判断も自然です。

対応:

- 引用符や `という表現` を使って、語を題材にしていることを明示する。
- 教材やルール説明の章では、狭い範囲に理由付き無視コメントを置く。

### 否定形に含まれる強い副詞

- 観測例: `この対策だけで問題が完全になくなるわけではない。` (`absolute-claim`)
- 観測例: `すべての読者に同じ効果が出るとは限らない。` (`absolute-claim`)

`完全に` や `すべて` は強い語ですが、否定形と組み合わさると、書き手はむしろ射程を狭めている場合があります。

対応:

- 文意が限定として読めるなら、そのまま残す。
- 誤読を避けたい場合は、`一部では残る`、`条件によって変わる` など、限定の内容を補う。

## `thin-sentence`

### `可能性がある` で終わるリスク要約

- 観測例: `予算が不足すると、来年度の実施時期が遅れる可能性がある。` (`thin-sentence`)
- 観測例: `利用者が増えるほど、問い合わせ対応の負荷が高まる可能性がある。` (`thin-sentence`)

`可能性がある` は、根拠が薄い文を閉じるためにも、不確実なリスクを慎重に扱うためにも使われます。前後で条件、対象、根拠が補われているなら、単独の文だけで薄いとは言い切れません。

対応:

- 条件や根拠を1語から1文だけ補い、読み手が判断できる材料を増やす。
- リスク列挙の章では、`thin-sentence` を参考指摘として扱う。

### 表現を説明している文

- 観測例: `「可能性がある」という表現は、根拠の弱さを隠すためにも使われる。` (`thin-sentence`)

表現そのものを説明している文では、対象語が出ること自体は自然です。本文の主張に曖昧さがあるのか、語の用法を説明しているだけなのかを分けて判断します。

対応:

- 引用符や `表現` という語で題材化を明確にする。
- ルール説明や教材の中では、理由付き無視コメントで残す判断を記録する。

## `placeholder`

### 指示詞としての `ここに`

- 観測例: `ここに判断の分岐点がある。` (`placeholder`)
- 観測例: `ここには、前提の違いが残っている。` (`placeholder`)

`ここに` は未記入欄の合図にも、直前の議論を受ける指示詞にもなります。後続に `分岐点`、`違い`、`課題`、`余地` などの概念名詞が来る場合は、文章構造の一部として機能していることがあります。

対応:

- 指示対象が直前で明確なら、そのまま残す。
- 読み手が迷いそうなら、`この条件に`、`この比較に` のように指示対象を具体化する。

### 未記入を論点として扱う文

- 観測例: `この表では、評価基準が未定義のまま残っている。` (`placeholder`)

`未定義` や `未定` は、未編集の印にも、問題点を説明する語にもなります。論点として使っている場合は、未完成箇所として機械的に消す必要はありません。

対応:

- `論点として`、`現時点では` などを添えて、未完成ではなく分析対象であることを明示する。
- 見出しだけでは誤読される場合、本文側で意図を補う。

## `excessive-parentheses`

### 略語展開や正式名称の密集

- 観測例: `教育評価委員会（EAC）は、年次報告書（2026年度版）で、評価指標（暫定）を示した。` (`excessive-parentheses`)

行政、研究、技術文書では、初出の機関名、略語、版数、注記を括弧で補う慣習があります。必要な補足でも、段落内で続くと視認性が落ちます。

対応:

- 初出だけ括弧で展開し、以降は略語に統一する。
- 長い補足は括弧の中に閉じ込めず、後続文へ分ける。
- 用語表がある文書では、本文側の括弧を減らす。

### 数値例や図表番号の挿入

- 観測例: `負荷の上限（例: 100MW）は、契約単位（サイトごと）で扱う。` (`excessive-parentheses`)

数値例、図表番号、注記が続くと、文としては成立していても見た目が詰まります。読者が括弧の階層を追う負担が増えるなら、構造で解くほうが安定します。

対応:

- 例示を別行や箇条書きに分ける。
- `options.maxPairs` を緩めるのは、文書ジャンルとして括弧が必要な場合に限る。

## `long-sentence`

### 参考情報を1文に詰めた行

- 観測例: `参考資料として、年次報告書、委員会議事録、補足データ、関連する利用規約、過去の検討メモを確認した。` (`long-sentence`)

参考欄や制作メモでは、文の論理が複雑でなくても、列挙だけで文字数が増えます。本文の主張なのか、単なる参照列挙なのかで判断が変わります。

対応:

- 参考情報は箇条書きに分解する。
- 本文の主張と出典メモを同じ文に入れない。
- 参照欄全体を別セクションに逃がす。

### 主体、根拠、対象、行動が1文に入る説明

- 観測例: `調査チームは、前年度の回答傾向と今年度の自由記述を照合し、授業改善に必要な観点を三つに整理した。` (`long-sentence`)

政策文書や研究メモでは、主体、根拠、対象、行動を1文で示すことがあります。情報が多い文は、それだけで悪いわけではありませんが、読者が途中で主語を見失うなら分割の候補です。

対応:

- 主節を先に置き、根拠や補足を後続文へ分ける。
- 1文に残す場合は、主語と述語の距離を短くする。

## `long-paragraph`

### 導入段落の集約

- 観測例: `この文書では、背景、問題設定、検討範囲、次の章で扱う論点を一つの段落にまとめて示す。` (`long-paragraph`)

導入で多くの役割を一段落にまとめると、読者は全体像を早くつかめる一方で、段落の負荷が高くなります。

対応:

- 背景、問題設定、構成案を分ける。
- 最初の段落は問いに絞り、章立ては次の段落で示す。
- 意図して集約する場合も、段落内の役割を増やしすぎない。

### 事実列挙の段落

- 観測例: `会議では、参加者数、実施時期、準備状況、未解決の論点、次回までの宿題が順に確認された。` (`long-paragraph`)

事実、根拠、対応を一段落に置くと、経緯は追いやすい一方で、後から参照しにくくなることがあります。

対応:

- 事実、判断、次の行動を段落で分ける。
- 後で参照する情報は箇条書きにする。

## `citation-needed`

### 直後に出典がある文

- 観測例: `近年、自治体のオンライン手続きは増えている。出典: 2026年度行政手続調査。` (`citation-needed`)

`近年` や `報告されている` は根拠を促す語ですが、直後に出典がある場合、文単位の検出だけでは過剰に見えることがあります。

対応:

- `2026年度行政手続調査では` のように、出典を文内へ組み込む。
- 出典ブロックが直後にある場合は、1行だけ理由付き無視コメントで抑制する。

### 数値で根拠化されている現在表現

- 観測例: `現在、登録者は約1.5万人で、前年度より2割増えている。` (`citation-needed`)

`現在` は時点依存の語です。数値があっても、いつの数値かが曖昧だと、後から読んだときに根拠が弱くなります。

対応:

- `2026年5月時点で` のように時点を固定する。
- 数値の出典が別にあるなら、同じ段落内に置く。

## `weasel-phrases`

### 出典が直前にある `とされている`

- 観測例: `調査報告書によると、この方式は小規模な運用で有効とされている。` (`weasel-phrases`)

`とされている` は出典不明に見えやすい表現です。ただし、直前で出典が明示されているなら、曖昧さは小さくなります。

対応:

- 出典名を文頭に置き、誰の見解かを明確にする。
- 出典がない場合は、`誰がそう述べているのか` を補う。

### 数値や条件を伴う限定表現

- 観測例: `この制度では、上限は申請額の2.5倍までとされている。` (`weasel-phrases`)

具体的な数値や条件がある場合、表現は曖昧な断定ではなく、制度文書の要約として機能していることがあります。

対応:

- 制度名や条項名を補う。
- 出典が不明なままなら、根拠確認の指摘として扱う。

## `repeated-connectors`

### 段落ごとの論理転換

- 観測例: `しかし、この条件では別の負担が生じる。` (`repeated-connectors`)

`しかし`、`一方で`、`つまり` などは、近い範囲で繰り返すと単調になります。ただし、章や段落ごとの論理転換として必要な反復もあります。

対応:

- 近接する接続語だけを見直す。
- 意味が同じなら片方を削る。
- 比較表や手順書では、接続語の統一を優先して残す判断もある。

### 要約の合図としての反復

- 観測例: `つまり、この変更は運用手順の見直しを求めている。` (`repeated-connectors`)

要約文の前に毎回 `つまり` を置くと読みやすい場合もありますが、同じ型が続くと文章のリズムが固定されます。

対応:

- `要するに`、`このため`、`言い換えると` などへ一部だけ変える。
- 見出しや箇条書きで要約の位置が明らかなら、接続語を削る。

## `actorless-action`

### 主体が前文にある行動表現

- 観測例: `委員会は改善案を確認した。今後、運用手順の見直しを進める。` (`actorless-action`)

後続文だけを見ると主体が薄く見えても、前文で主体が明示されている場合があります。読み手が誰の行動か迷わないなら、修正が要らない場合もあります。

対応:

- 主体が前文から自然に引き継がれるか確認する。
- 読み手が迷うなら、`委員会は`、`事務局は` などを補う。

### 資料の方針を要約する文

- 観測例: `資料では、来年度に向けて試行範囲を広げる方針が示されている。` (`actorless-action`)

資料や報告書を要約する場合、実際の行動主体が文中に出ないことがあります。責任所在を明確にしたい文書では、誰が示した方針なのかを補うほうがよいです。

対応:

- `事務局資料では`、`審議会は` のように発行者や主体を補う。
- 出典の要約であることが十分に明確なら、理由付きで残す。

## 無視コメントの判断基準

`nihongo-slopless` の指摘を消すことは、文章の品質を保証しません。指摘は編集候補であり、書き手が「修正する」「残す」「条件付きで残す」のどれを選ぶかが本来の判断です。

無視コメントは、修正しない判断を文書内に残すための装置です。件数を下げるためのスイッチとして広く使うと、後から文書の意図を追いにくくなります。

置くかどうかの目安は、書き手が「なぜそのまま残したか」を第三者に説明できるかです。説明できる理由には、引用本文を原文どおりに保つこと、専門用語の定型表現を尊重すること、教育目的で例として残すこと、章単位で文体を一貫させること、検証データの一部として残すことなどがあります。

理由付きの無視コメントは、`<!-- nihongo-slopless-ignore <rule>: 理由 -->` の形式で、コメントの直後行だけを対象にします。複数行や章単位で抑制したい場合は、`disable` と `enable` のペアを置き、その範囲を狭く保ちます。

迷ったときは、先に小さな修正案を書いてみてください。文章の意図が変わらず、対象や根拠が前より明確になったら採用します。意図が変わるか冗長になるなら、理由付きで残す判断に戻します。

このツールは、書き手を裁くためではなく、編集の余地を見つけるために動きます。判断をツールに委ねず、どの指摘を修正し、どの指摘を残すことが文章の責任になるかを人間が決めてください。
